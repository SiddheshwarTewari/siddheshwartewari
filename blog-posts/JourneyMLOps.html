My Journey Learning ML Ops as an AI Student

6 min read By Sid Tewari

I remember the pride I felt after training my first "production-ready" model in university. It was a convolutional neural network for image classification. The accuracy on the test set was a respectable 95%. I saved it as final_final_model_v3.pkl, sent the file to my professor, and considered my job done.

Then, a simple question shattered that illusion: "Great, but how would someone actually use this?".

I had no answer. My "production-ready" model lived on my laptop, trapped in a pickle file, completely useless to anyone else. That moment was my first, clumsy step into the vast and chaotic world of Machine Learning Operations (ML Ops).

The University Curriculum vs. The Real World

My own experiences and studies in AI gave me an incredible foundation. I became proficient in all aspects of AI, from math to supervised/unsupervised learning to modeling using Scikit-learn, TensorFlow, and Pytorch.

However, the University curriculum, my own internships/experience, and practice ended at the evaluation metric, I never dived deep into what comes next.

As I thought about this, questions began to raise in my mind, such as "How do you serve this model to millions of users?" or "How do you ensure the model remains accurate as real-world data changes?"

MLOps is the answer to all these questions. It's the engineering discipline of taking ML models from a Jupyter Notebook to a live, reliable, and scalable service.

My Self-Taught MLOps Curriculum: The Four Pillars

I began structuring my own learning. I broke MLOps down into four key pillars that became my personal syllabus.

  • Packaging and Deployment: From .pkl to API
  • Versioning: More Than Just final_v3.pkl
  • Monitoring: The Model is Deployed. Now What?
  • The CI/CD Pipeline: Automation is Key

Packaging and Deployment

The first step was learning how to free my model from my laptop. I learned about:

  • Web Frameworks: Building simple REST APIs with FastAPI and Flask. Suddenly, my model could be accessed over the internet. A simple POST request with data would return a prediction.
  • Containers: Packaging my API and model into a Docker container. This became the standardized unit of deployment I could ship anywhere.

Versioning

I quickly realized that final_v3.pkl was a terrible naming convention. I needed to version everything:

  • Model Versioning: Tools like MLflow or DVC (Data Version Control) helped me track which model version (and its associated metrics) was deployed.
  • Data Versioning: Often overlooked. I used DVC to version my training datasets, so I could always reproduce an old model exactly.

Monitoring

Deploying the model wasn't the finish line; it was the starting line. I built a simple dashboard to track:

  • Latency: How long does it take to get a prediction? Users won't wait 10 seconds.
  • Throughput: How many predictions per second can my API handle?
  • Data Drift: Is the live data starting to look different from the data I trained on? This is a silent killer of model accuracy.
  • Model Performance: In projects where I could get ground truth data later, I tracked accuracy over time to see if the model was decaying.

The CI/CD Pipeline

The final piece was automation. I used GitHub Actions to create a simple pipeline:

  • On a Git push, it would automatically run my tests.
  • If tests passed, it would build a new Docker image.
  • It would then deploy the new image to a cloud service like Heroku.

Key Takeaways for Other Students

  • Start Small: You don't need Kubernetes. Start by turning a simple Scikit-learn model into a local API with Flask.
  • Embrace DevOps: MLOps is fundamentally DevOps for machine learning. Learn the basics of Docker, Git, and CI/CD. They are force multipliers.

Conclusion: The Important Lesson

My journey into MLOps taught me that the most important metric isn't accuracy; it's impact.

A 95% accurate model that is deployed, reliable, and used by people is infinitely more valuable than a 99% accurate model trapped in a notebook. As I finish my degree and look towards my career, I'm no longer just a student of machine learning. I'm learning to be an engineer who builds systems that learn.
